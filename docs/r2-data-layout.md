# R2 Data Layout

Arktrace uses Cloudflare R2 (S3-compatible) to distribute pre-built pipeline
artifacts and to cache public external source files so that CI and local
pipeline runs do not have to re-download them each time.

All paths below are relative to the root of the configured bucket
(`S3_BUCKET`).

---

## Bucket layout

```
<bucket>/
  public/
    arktrace/
      raw-cache/              ← public external source files, cached here so
      │                          CI and local runs skip re-downloading
      │
      ├── gdelt/
      │   ├── latest          ← plain text: "20260412"  (YYYYMMDD)
      │   └── 20260412.zip    ← 14-day GDELT CSV bundle (data/raw/*.CSV)
      │
      ├── sanctions/
      │   ├── latest          ← plain text: "20260408"  (YYYYMMDD)
      │   └── 20260408.zip    ← opensanctions_entities.jsonl
      │
      └── marine-cadastre/
          └── 2024.zip        ← annual AIS track archive (pushed manually)
                                 see "Marine Cadastre" section below
      common/                 ← artifacts shared across all regions
      │
      ├── gdelt.lance.zip     ← Lance vector store built from raw-cache/gdelt/
      │                          ~1.2 GB; updated when GDELT is re-ingested,
      │                          not on every pipeline run
      └── validation_metrics.json
                              ← scoring validation metrics (precision/recall)

      regional/               ← one sub-directory per region
      │
      ├── singapore/
      │   ├── latest          ← plain text: "20260412T120000Z"
      │   ├── 20260412T120000Z.zip
      │   └── 20260410T153012Z.zip   ← previous generation (kept N=3)
      │
      ├── japan/
      ├── middleeast/
      ├── europe/
      └── gulf/
          (same structure as singapore/)
```

Each **regional zip** contains everything the dashboard needs to serve one
region:

| File | Description |
|---|---|
| `{region}.duckdb` | AIS positions, vessel features, scoring results |
| `{region}_watchlist.parquet` | Ranked candidate watchlist |
| `{region}_causal_effects.parquet` | C3 causal sanction-response model output |
| `{region}_graph/` | Lance Graph ownership chains (nodes + edges) |

## Data classification

### By generation method

| Artifact | Generated by | Why |
|---|---|---|
| `{region}.duckdb` | **CI** | Built from seeded synthetic vessels + public data; no live AIS key required |
| `{region}_watchlist.parquet` | **CI** | Scoring pipeline output over the DuckDB above |
| `{region}_causal_effects.parquet` | **CI** | Causal model output over the DuckDB above |
| `{region}_graph/` | **CI** | Ownership graph built from vessel_registry seeded from DuckDB |
| `validation_metrics.json` | **CI** | Computed from labeled watchlist; no external data required |
| `gdelt.lance` | **CI** | Built from raw-cache/gdelt/ (no API key, public HTTP) |
| `raw-cache/gdelt/` | **CI** | GDELT CSV files from `data.gdeltproject.org` (public, no key) |
| `raw-cache/sanctions/` | **CI** | OpenSanctions JSONL from `data.opensanctions.org` (public, no key) |
| `raw-cache/marine-cadastre/` | **Manual push** | Large annual archive; pushed once per year by a maintainer |
| Live AIS in DuckDB | **Local + upload** | Requires `AISSTREAM_API_KEY`; CI uses synthetic seed instead |

### By scope

| Artifact | Scope | Notes |
|---|---|---|
| `gdelt.lance` | **Common** | Single GDELT corpus queried by all regions |
| `validation_metrics.json` | **Common** | Aggregate metrics across all regions |
| `raw-cache/*` | **Common** | Source files shared across all region builds |
| `{region}.duckdb` | **Regional** | Separate per region |
| `{region}_watchlist.parquet` | **Regional** | |
| `{region}_causal_effects.parquet` | **Regional** | |
| `{region}_graph/` | **Regional** | Ownership chains differ by region vessel pool |

## CI workflow

The `data-publish` workflow (`.github/workflows/data-publish.yml`) runs the
full multi-region pipeline on public data and publishes the results to R2.

**Steps:**

1. **Pull raw cache** — download `raw-cache/gdelt/` and `raw-cache/sanctions/`
   from R2 if they are fresher than the CI workspace copy; skip re-downloading
   from the original sources.
2. **Run pipeline** — `scripts/run_public_backtest_batch.py` for all five
   regions (`--stream-duration 0 --seed-dummy`).
3. **Push regional zips** — `sync_r2.py push --keep 3` uploads one zip per
   region under `regional/{region}/` and updates the `latest` pointer.
4. **Push common artifacts** — `validation_metrics.json` and, when GDELT was
   re-ingested, `common/gdelt.lance.zip`.
5. **Update raw cache** — `sync_r2.py push-cache` uploads updated GDELT and
   sanctions files so the next CI run can skip the download.

Trigger: manual dispatch (`workflow_dispatch`) or automatically after
`Public Backtest Integration` succeeds on `main`.

## User pull options

Use `scripts/sync_r2.py` to pull data locally before running the dashboard.

```bash
# Pull one region (minimum download — recommended for most users)
uv run python scripts/sync_r2.py pull --region singapore

# Pull a different region
uv run python scripts/sync_r2.py pull --region japan

# Also pull the GDELT Lance store (~1.2 GB) for brief/chat context
uv run python scripts/sync_r2.py pull --region singapore --with-gdelt

# Pull raw source cache to run the pipeline locally without re-downloading
uv run python scripts/sync_r2.py pull-cache --source gdelt
uv run python scripts/sync_r2.py pull-cache --source sanctions
```

After pulling, start the dashboard:

```bash
bash scripts/run_app.sh --region singapore
# or, with Docker:
docker compose up
```

### Approximate download sizes

| Command | Approx. size |
|---|---|
| `pull --region <region>` | 50–200 MB (varies by region vessel count) |
| `pull --with-gdelt` | +1.2 GB |
| `pull-cache --source gdelt` | ~200 MB (14-day CSV bundle) |
| `pull-cache --source sanctions` | ~500 MB |
| `pull-cache --source marine-cadastre` | Multi-GB (annual, rarely needed) |

## Credentials

R2 credentials are read from environment variables (or `.env`):

| Variable | Description |
|---|---|
| `S3_BUCKET` | Bucket name (e.g. `arktrace-data`) |
| `S3_ENDPOINT` | `https://<account_id>.r2.cloudflarestorage.com` |
| `AWS_ACCESS_KEY_ID` | R2 Access Key ID |
| `AWS_SECRET_ACCESS_KEY` | R2 Secret Access Key |
| `AWS_REGION` | `auto` |

These credentials are **only** used by `sync_r2.py` and `bootstrap.py` for
push/pull. The dashboard itself always reads from local disk — setting these
variables does not route app reads to R2 (see `USE_S3` in
[Deployment](deployment.md) for the opt-in S3 serving mode).

For **read-only public pulls** (end users running the dashboard), no
credentials are needed if the bucket's `public/` prefix has public access
enabled (Cloudflare R2 → Settings → Public Access).

### CI secrets

The `data-publish` workflow reads the following repository secrets:

| Secret | Maps to |
|---|---|
| `R2_BUCKET` | `S3_BUCKET` |
| `R2_ENDPOINT` | `S3_ENDPOINT` |
| `R2_ACCESS_KEY_ID` | `AWS_ACCESS_KEY_ID` |
| `R2_SECRET_ACCESS_KEY` | `AWS_SECRET_ACCESS_KEY` |

## Marine Cadastre (manual push)

Marine Cadastre annual AIS track archives are large (multi-GB per year) and
only need to be refreshed once a year. They are pushed manually by a
maintainer, not by CI.

```bash
# Download the archive for a given year
uv run python -m src.ingest.marine_cadastre --year 2024 --output data/raw/marine_cadastre

# Zip and push to R2
zip -r marine-cadastre-2024.zip data/raw/marine_cadastre/2024/
aws s3 cp marine-cadastre-2024.zip \
    s3://${S3_BUCKET}/public/arktrace/raw-cache/marine-cadastre/2024.zip \
    --endpoint-url ${S3_ENDPOINT}
```

Once cached, the pipeline picks it up via:

```bash
uv run python scripts/sync_r2.py pull-cache --source marine-cadastre --year 2024
```

## Proprietary data

Live AIS data collected via AISstream.io, AISHub, or Global Fishing Watch
(GFW) is **not** published to the public R2 prefix. If you have collected
your own live AIS data and want to share it within your team:

1. Push your regional DuckDB (which contains the live positions) to a
   **private** bucket or a private prefix within the same bucket.
2. Share credentials only with authorised team members.
3. Do not push to `public/arktrace/` — that prefix is reserved for OSS
   artifacts that can be freely distributed.
